Project Overview

Intent Stabilization via Selective Resampling

01 / 14

The project goal is to make any instruction-tuned model understand user intent before it answers: sample multiple intent hypotheses, find unstable stages, resample only the weak point, and ask a clarification question only if uncertainty stays high.

Motivation

Why this project exists

02 / 14

Answer-too-early failure

The request is ambiguous, but the model answers under an unstated assumption and gets the task wrong.

Clarify-too-early failure

The model detects uncertainty but turns every ambiguous request into a reflexive clarification turn.

Wrong confidence target

Typical self-confidence conflates confidence about the answer with confidence about the user's intent.

Project objective

maximize expected task success - lambda_q * clarification_cost - lambda_c * extra_model_calls

The core design choice is to spend extra compute only where intent uncertainty is concentrated, rather than rerunning the whole reasoning pipeline or clarifying immediately.

Research questions

  • Can a simpler repair-pass resampling controller beat targeted clarification on ambiguous-intent tasks?
  • Can structured agreement serve as a useful intent-confidence proxy without token-logprob tracing?
  • Can a smaller student recover part of the teacher-wrapper gain via LoRA SFT?

Method

resample_clarify: repair pass before clarification

06 / 14
Stage 1
Sample ambiguity

Generate multiple structured ambiguity assessments at moderate temperature.

Stage 2
Sample intent models

Enumerate candidate interpretations, missing variables, and dominant hypotheses.

Stage 3
Aggregate agreement

Score agreement per stage and detect whether the latent-intent state is still unstable.

Stage 4
Repair the full state

Refresh the ambiguity-intent-strategy bundle once when intent remains unstable, rather than chasing a single weak stage.

Stage 5
Act

Answer, narrow, present alternatives, or ask a generic clarification if intent is still unstable.

Current selected dev configuration

initial samples K = 3 repair samples = 1 max rounds = 1 clarification fallback threshold = 0.80

This was selected by early stopping from the first winning InfoQuest-dev configuration: k3_r1_c08.

What is intentionally out of scope for v1

  • no token-logprob-based trace analysis
  • no full-pipeline resample-until-confident loop
  • no RL or GRPO in the paper-critical path
  • no retrieval dependence in the main method
The method is agreement-based and modular Speaker cue: 55s

Execution

How the project runs efficiently end to end

09 / 14

Runtime strategy

  • Tinker-native inference only. No local GPU dependency.
  • Sharding happens at dataset/model leaves and teacher-rollout shards.
  • Aggregation happens only after every shard manifest is complete.
  • Expensive phases are resumable at the shard and method level.
Sequential orchestration: ./.venv/bin/python scripts/run_research_pipeline.py --config ... Parallel command planner: ./.venv/bin/python scripts/plan_jobs.py --config ... --phase ... Leaf execution: ./.venv/bin/python scripts/run_sharded_leaf.py --config ...

Recommended operator workflow

control
Fit config and aggregate

Create selected_config.yaml, aggregate matrix shards, export audit, regenerate paper tables.

gate windows
Test gate shards

Run qwen3_30b test shards first so you can decide early whether to spend on training.

ablation windows
Background controls

Keep ablations running in parallel, but do not wait on them before the gate decision.

post window
Teacher -> student later

Only reuse the free windows for rollouts and SFT after Phase 1 is worth continuing.

See docs/end_to_end.md, docs/job_parallelism.md, docs/training.md, and docs/paper.md.

Diagnosis

What the current evidence says went wrong

11 / 14

Primary failure mode

Established
  • The original selective weak-point controller over-spent compute.
  • It still clarified on almost every example instead of resolving intent.
  • So the intended behavior, “repair then avoid clarification,” did not materialize.

What the partial ablations suggest

Still finishing
  • Single-sample behavior is weak, so multi-sampling is still useful.
  • The full-repair family looks stronger than the selective-repair family.
  • The generic clarification fallback looks better than the targeted fallback.
  • Calibration is not the main bottleneck.
Operational constraint

What slowed the live runs

Tinker jobs are fine under normal resumable execution, but laptop sleep / lid-close events poison live sessions. The fix is operational: keep the machine awake, use retry wrappers, and treat sleep-resume as a restart event, not a recoverable network blip.

Research diagnosis: the first controller was too complicated and too eager to clarify anyway Speaker cue: 35s
