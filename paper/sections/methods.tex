\section{Methods}

The evaluation compares five inference-time methods over the same Tinker-hosted task models.

\begin{itemize}[leftmargin=1.5em]
\item \textbf{direct\_answer}: answer immediately without ambiguity handling.
\item \textbf{generic\_hedge}: answer directly with hedging language.
\item \textbf{generic\_clarify}: always ask a generic clarification question.
\item \textbf{targeted\_clarify}: detect ambiguity, model candidate interpretations, choose a response strategy, and continue to a final answer after follow-up when appropriate.
\item \textbf{resample\_clarify}: sample multiple ambiguity and intent hypotheses, score agreement, selectively resample the weakest stage, calibrate intent confidence, and clarify only when uncertainty remains above threshold.
\end{itemize}

For the student phase, the repository exports teacher rollouts into stage-level SFT records and preference pairs. The intended training path is modular: the student learns the same structured stage outputs used by the teacher wrapper, so the distilled model plugs into the existing runtime without changing the pipeline architecture.

\paragraph{Workflow phases.}
The end-to-end workflow is intentionally gated. Phase~1 develops and evaluates the teacher method: frozen data preparation, a development sweep on InfoQuest-dev, calibrator fitting, a fresh test-set gate on the primary teacher model, and selective-resampling ablations. Phase~2 begins only if that gate is strong enough: teacher rollout export on InfoQuest-train, LoRA SFT for the 4B student, and student evaluation on both held-out test sets. Phase~3 is optional: if the SFT student is promising, the repository can also prepare preference data and a DPO configuration for a later refinement pass.
